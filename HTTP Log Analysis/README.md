# **Splunk Basics – HTTP Log Analysis**
##  **Objective**

The aim of this lab is to understand how HTTP logs can be analyzed using Splunk.

In this lab, you will:

* Learn how to upload and analyze HTTP logs in Splunk
* Identify client-side and server-side errors
* Detect suspicious web activity
* Analyze large data transfers and unusual URI access patterns

##  **Lab Setup**

### **Requirements**

* Splunk installed and accessible through Splunk Web
* HTTP logs in JSON format (Zeek-style)
* Basic understanding of Splunk Search Processing Language (SPL)

---

### **Data Source**

* **Log Type:** Zeek-style HTTP logs
* **Format:** JSON
* **File Name:** synthetic_zeek_http.json

The log file should be stored in a location that is accessible during the upload process in Splunk.

##  **Steps to Upload HTTP Logs into Splunk**

1. Open Splunk Web in your browser

2. Navigate to:
   **Settings → Add Data**

3. Select the **Upload** option

4. Choose the file:
   **synthetic_zeek_http.json**

5. Configure input settings:

   * **Source Type:** json
   * **Index:**

     * Use default index (**main**)
     * or create a new index (**http_lab**)

6. Complete the upload process

7. Verify that the logs are successfully indexed by running a search

 **Lab Tasks**

The following tasks are performed using SPL queries to analyze HTTP traffic.

### ✅ **Task 1: Identify Top Sources Generating Web Traffic**

This task helps identify the IP addresses that are generating the highest number of HTTP requests.

```id="xq6k6k"
index=http_lab sourcetype="json"
| stats count by id.orig_h
| sort -count
| head 10
```

### **Explanation**

* `id.orig_h` represents the client/source IP address
* This query displays the top 10 IPs generating the most web traffic

---

### ✅ **Task 2: Analyze Server-Side Errors (5xx)**

This task focuses on identifying server-side errors.

```id="n0n7p6"
index=http_lab sourcetype="json" status_code>=500 status_code<600
| stats count as server_errors
```

### **Explanation**

* Status codes between **500–599** indicate server errors
* Useful for identifying backend issues or misconfigurations

---

### ✅ **Task 3: Detect Suspicious User-Agents**

This task is used to identify requests generated by automated tools or scripts.

```id="6q4d7o"
index=http_lab sourcetype="json"
user_agent IN ("sqlmap/1.5.1","curl/7.68.0","python-requests/2.25.1","botnet-checker/1.0")
| stats count by user_agent
```

### **Explanation**

These user-agents are commonly associated with automated or testing tools:

* **sqlmap** – Used for SQL injection testing
* **curl** – Command-line HTTP client
* **python-requests** – Used in scripts and automation
* **botnet-checker** – May indicate bot-related activity

---

### ✅ **Task 4: Identify Large Data Transfers**

This task helps detect HTTP responses with large data sizes.

```id="m7r4g3"
index=http_lab sourcetype="json"
resp_body_len>500000
| table ts id.orig_h id.resp_h uri resp_body_len
| sort -resp_body_len
```

### **Explanation**

* `resp_body_len` represents the size of the response in bytes
* Large transfers may indicate:

  * File downloads
  * Data exfiltration
  * Delivery of large payloads

##  **Additional Practice**

You can extend the analysis using the following queries:

---

### 🔹 **Most Requested URIs**

```id="x4b9s2"
index=http_lab sourcetype="json"
| stats count by uri
| sort -count
```

---

### 🔹 **Client-Side Errors (4xx)**

```id="k2g1z8"
index=http_lab sourcetype="json" status_code>=400 status_code<500
| stats count by status_code
```

---

### 🔹 **Rarely Accessed URIs**

```id="z9m3v1"
index=http_lab sourcetype="json"
| stats count by uri
| where count=1
```
